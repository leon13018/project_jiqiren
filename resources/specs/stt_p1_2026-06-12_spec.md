# STT Phase 1 — Deepgram 串流基礎（播完才聽）— SDD spec

> 統領設計：`resources/specs/stt_bargein_2026-06-12_design.md`（commit a3b0dd3，已核可）。
> 本 spec = Phase 1 範圍（WHAT）；step-by-step 見 `resources/plans/stt_p1_2026-06-12_plan.md`（HOW）。

## 1. 背景與動機

統領設計 §3 Phase 1：先建立「音訊鏈 → Deepgram → input queue」的完整路徑並實機驗證延遲 / 繁體 / NLU 命中率，barge-in（25% 開麥、interrupt、preempt）全部留 Phase 2。本 Phase 的 arm 時機 = TTS 播完後（`read_customer_input` 既有 `wait_idle()` 之後），與現行 UX 完全相同——顧客聽完提示句才開麥。

## 2. 設計核心與行為規約

### 2.1 與統領設計的差異（精化，共兩處）

1. **無 asyncio**：統領設計 §2.5 寫 SttWorker 用「daemon thread + 常駐 asyncio loop（tts.py 同 pattern）」。實作探勘後改為 **`websockets` 同步 client（v12+ `websockets.sync.client`）+ arm 時起 session threads，無 asyncio、無常駐 thread**。理由：tts.py 用 asyncio 是被 edge-tts async-only API 綁定；Deepgram 串流用同步 client 即可，少一整層 asyncio-in-thread 的 race 面（threading reference Part A 地雷區），延遲不變。
2. **arecord 死亡不自動重啟**：統領設計錯誤矩陣寫「EOF 偵測自動重啟（有限次）」。改為 **session 直接結束、下次 arm 重建**——arecord 改為每 session 一條後，session 壽命=單次顧客等待窗，窗內重啟收益趨近零；等待 timeout 走既有 reprompt、下次 arm 自然重生。少一個重啟計數狀態。

統領設計其餘規約不變。

### 2.2 SttWorker 生命週期

```
arm()（冪等；_lock 保護）
  ├─ 已 armed / 已停用 → no-op
  ├─ 無 api_key → 印一次警告 + 本次執行停用（鍵盤照常）
  ├─ ws 連線（失敗重試 1 次；401 → 印警告 + 本次執行停用；其他失敗 → 本輪放棄、下次 arm 再試）
  └─ 成功 → 起 audio source（arecord subprocess）+ 2 條 daemon thread：
       SttSender：audio.read(3200B=100ms) → ws.send，EOF / stop 即止
       SttReceiver：ws.recv → JSON → type=Results 且 speech_final=true →
                    transcript 去頭尾標點 → 非空 → print [語音辨識] + sink(text)
disarm()（冪等）
  └─ stop event + audio.close()（terminate arecord）+ ws.close() + join(timeout=1)
shutdown() = disarm()（main.py finally 鏈）
```

- session 期間多個 speech_final 各自注入；input queue 既有 latest-wins drain 語意自然處理殘留。
- arecord **每 session 一條**（arm 起、disarm 殺）——避免持續收音塞爆 pipe buffer 造成下輪殘留舊音訊。
- 例外處理：sender / receiver 內 Exception 結束該 session（receiver 在非 disarm 觸發時印 `[語音辨識] ⚠️ 串流中斷`）；不自動重連（下次 arm 重建）。

### 2.3 Deepgram 連線參數（統領設計 §2.5 既定值）

```
wss://api.deepgram.com/v1/listen?model=nova-3&language=zh-TW&encoding=linear16
    &sample_rate=16000&channels=1&interim_results=true&endpointing=300&smart_format=false
header: Authorization: Token <DEEPGRAM_API_KEY>
```
- key 來源：環境變數 `DEEPGRAM_API_KEY`（Pi 端設定；絕不進 repo）。
- arecord 指令：`arecord -q -f S16_LE -r 16000 -c 1 -t raw`（+ 可選 `-D $STT_ARECORD_DEVICE`，環境變數未設則用 ALSA 預設裝置）。
- Pi 首測若 handshake 回 400（language 代碼不被該 model 接受）→ 改試 `language=zh-Hant`，結果回寫本 spec（living document）。

### 2.4 正規化

`_normalize_transcript(text)`：去**頭尾**標點與空白（`。，、！？；：．,!?;:` + whitespace），句中不動。動機：strict-short 比對（`好。`≠`好`）。全形數字 / 控制字元 / 長度上限由 `read_customer_input` 既有 `normalize_input` 統一處理（注入文字與鍵盤輸入同路徑過它，不重複實作）。

### 2.5 注入點與 arm/disarm 佈線

- `input_reader.InputReader.inject(text)`＋module-level `inject(text)`：`self._q.put(text)`——STT 與鍵盤共用單 queue，producer 端零分流。
- `main.py read_customer_input`：`tts.wait_idle()` 之後 `stt.arm()`，read 迴圈包 `try/finally` 確保 `stt.disarm()`（拿到輸入、timeout、`q` 退出三路徑皆 disarm）。
- `read_terminal_key`（商家層）/ `sleep`（L5）**不** arm。
- `main()` finally 的 shutdown 鏈 tuple 加 `"stt"`（居首：先停收音）。
- stt module 採 **lazy singleton**（`_get_worker()` 首次 arm 時建）：與 input_reader eager singleton 不同——import 時不需任何 thread / 不讀 env，Windows pytest import 零副作用。
