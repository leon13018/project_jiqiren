# perf_w5 — TTS 內容定址快取＋預熱（spec — WHAT）

> perf campaign 追加波（user 2026-06-12 提案：固定語音預先合成，效能極致化）。
> 裁決：架構 C（快取＋預熱＋資產進 git）＋預熱範圍「固定句＋小域模板變體」。
> 可行性根基：rate 由 `len(text)` 確定、VOICE 固定 → **同文字永遠產生同音訊**。

## 1. 背景與動機

業務對話語音絕大多數是字面常數（l1~l5_text／shared／HAWK_SLOGANS／confirm 文案），
動態的只有模板插值（商品 ×2、單位 ×2、數量、金額）。現行每句都走 edge-tts 雲端合成
（Pi 上 ~0.5–2s）；w2 的 prefetch 只能把合成藏進播放等待。**內容定址快取讓固定句的
合成直接消失**（零合成零網路），且資產進 git 後 **demo 現場斷網也能播全部固定語音**
——「快」與「穩」同時極大化。

副作用簡化：每段文字有自己的雜湊檔名 → w2 的共用雙 buffer（`_TTS_BUF_PATHS`／
`_buf_idx`／`_next_buf`）**退役**，不存在互踩問題；prefetch 簡化為「下一句沒快取就
先合成進快取」。

## 2. 設計核心與行為規約

### 2-0. 不變式（沿用 w2 spec §2-0 五條，全部繼續成立）

FIFO／`_pending` 與 wall-clock budget 起算點／print 時序／stdin=DEVNULL／ALSA drain／
失敗 noisy 字面／`wait_idle` 30s／shutdown 順序／基底零改動。
新增一條：**快取永不含半寫檔**——合成先寫 `<cache>.tmp` 再 `os.replace` 原子搬移；
中斷殘留的只會是 `.tmp`（下次重合成覆寫），`.mp3` 一旦存在必完整。

### 2-1. 內容定址快取（`tts.py`）

```python
import hashlib
import os
（既有 import 區補這兩個）

# perf_w5：內容定址快取目錄——package-anchored（非 cwd 依賴），Pi 上隨 git pull 取得
# 預熱資產；執行期合成的動態句也存入此處自我增長（重啟後仍有效，SD 卡非 tmpfs）。
# 模組變數＝測試 seam（monkeypatch 指到 tmp_path）。
_CACHE_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), "tts_cache")


def _cache_path_for(text: str) -> str:
    """內容定址：同文字（同 VOICE＋rate）永遠同檔名。任一合成參數改變 → key 變 → 自然失效。"""
    key = f"{VOICE}|{_pick_rate(text)}|{text}".encode("utf-8")
    return os.path.join(_CACHE_DIR, hashlib.sha1(key).hexdigest() + ".mp3")


def _store_into_cache(tmp_path: str, cache_path: str) -> None:
    """tmp → cache 原子搬移（防中斷殘留半寫檔進快取）。tmp 缺失時 no-op——
    測試以不寫檔的 fake _synthesize 注入，此 seam 讓既有 fake 全數免改。"""
    if os.path.exists(tmp_path):
        os.replace(tmp_path, cache_path)
```

`on_thread_start` 末尾補 `os.makedirs(_CACHE_DIR, exist_ok=True)`。

### 2-2. `_process` 改造（雙 buffer 退役）

**階段 1（三層 fallback）**：prefetch 標記命中 → 用其路徑；否則快取存在 → 用快取
（**零合成**）；否則合成到 `<cache>.tmp` → `_store_into_cache` → 用快取路徑。
合成失敗 noisy print＋return 字面不變。

**階段 2.5（prefetch 簡化）**：peek 下一句 → 快取存在 → 標記 `(nxt, cache_path)`
（瞬時預取）；不存在 → 合成入快取＋標記；失敗清標記靜默（理由註解沿用 w2）。

**移除**：`_TTS_BUF_PATHS`、`_buf_idx`、`_next_buf`。`_prefetched` 標記保留
（測試 seam：fake synth 不寫檔時仍能表達「已預取」；真實環境＝快取路徑快捷）。
`_peek_next` 不變。

### 2-3. 預熱腳本（新檔 `myProgram/tts_prewarm.py`）

`python3.11 -m myProgram.tts_prewarm`（Pi 端跑，需網路）：

- **自動枚舉**：`l1_text`～`l5_text`、`shared` 六個文案模組的公開 `str` 常數，
  排除含 `"{"` 的模板。（keyword 清單在 keywords.py 是比對資料非語音，不掃；
  少數 print 專用文案被多錄是無害的幾百 KB，換取枚舉邏輯零維護。）
- **手列小域變體**：`HAWK_SLOGANS` 全部＋對每個 `PRODUCTS`：
  `QTY_PROMPT_TEMPLATE`／`QTY_CLARIFY_TEMPLATE`／`AT_CAP_NOTICE_TEMPLATE` 插值。
- 去重保序 → 逐句：已在快取 skip；否則 `asyncio.run(_synthesize)` 到 tmp →
  `_store_into_cache`。逐句印結果，結尾印 `N 新合成 / M 已存在 / K 失敗`。
- 枚舉邏輯抽成純函式 `_prewarm_texts() -> list`（可單測）。

### 2-4. 資產進 git（bootstrap，pineedtodo 指示 user 執行一次）

1. Pi 端跑 prewarm → 產出 `myProgram/tts_cache/*.mp3`（估 60–100 句 × ~30KB ≈ 2–3MB）。
2. Pi 端 `git add myProgram/tts_cache`＋commit＋push（一次性 bootstrap；
   Pi 端 git 操作屬 standard-workflow 允許範圍）。
3. 之後文案改動 → 重跑 prewarm → 新 hash 檔同法補 commit；
   舊孤兒 mp3 留檔無害（watch-list：累積多了再做清理工具）。
