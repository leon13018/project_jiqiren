# STT Phase 2 — 讀 ch0 + 預熱連線（turn-taking polish）spec

> 2026-06-16 定案。承接統領設計 `stt_bargein_2026-06-12_design.md`：原 Phase 2 = barge-in 搶話，**經 AEC 實測證實在本硬體不可行**（見 §1），改為「快速輪流」品質/延遲微調。

## 1. 背景與動機

統領設計原規劃 Phase 2 為真 barge-in（邊播邊聽、講一半打斷），前提是 ReSpeaker XVF-3000 硬體 AEC 能消掉機器人自聲。**2026-06-16 在 Pi 上實測，barge-in 前提不成立**：

| 測法 | 結果 |
|---|---|
| 硬體 AEC（XVF-3000，6ch 韌體 + 參考 ch5 確認在位 + AEC 參數全啟用） | 消除 ~0–2 dB |
| 最佳線性 AEC（離線最小平方 FIR，任何線性消除器的數學上限） | 7.4 dB（廉價喇叭）/ 9.4 dB（主動式好喇叭） |
| 1 秒短窗 vs 整段 6 秒 ERLE | 8–11 dB ≈ 整段 → **排除時脈漂移**（自適應 AEC 也救不了多少） |
| 正規化互相關 norm_xcorr | 0.178（極低）→ 回聲與參考**線性關係極弱、大量非線性** |

真 barge-in 通常需 20–30+ dB 消除；本硬體封頂 ~10 dB（殘留 ~30% 振幅，雙講分離不可靠）。瓶頸是**喇叭/麥近距聲學耦合 + 房間殘響 + 喇叭非線性**——換喇叭僅 +2 dB 證明非設備主因。**結論：barge-in 收掉**；要真搶話需重新設計裝置聲學或上神經 AEC（皆研究級、非本期）。

**轉而做兩個 de-risk 過程浮現的真實改進**（turn-taking，不需 AEC）：
- **① 讀 ch0**：裝置是 6-channel 韌體，現況 STT 用 `arecord -c 1` 在 6ch 裝置上被 plughw **降混 6 軌**（ch0 處理後被 ch1–4 原始麥稀釋 ~5/6）。改成只讀 **ch0（晶片波束成形 + 降噪後）** → 吵雜展場辨識更乾淨。
- **② 預熱連線**：現況 `read_customer_input` 是 `tts.wait_idle()`（全播完）才 `stt.arm()`，開麥瞬間還要等 Deepgram ws 連線（~200–300ms）才真的在聽。改成 TTS 播放期間就先連好（但丟棄聽到的機器人聲），播完瞬間 go-live → 省掉那段等待。

## 2. 設計核心 + 行為規約

兩增量獨立、可分開 commit / merge（① 先上 demo 即受益，② 後上不卡 ①）。

### 增量 1 — 讀 ch0（程式內反交錯）
- `arecord` 由 `-c 1` 改 `-c 6`（裝置須為原生 6ch；Pi 端 `STT_ARECORD_DEVICE=hw:CARD=ArrayUAC10` → pineedtodo）。
- 新增純函式 `_extract_ch0(buf)`：交錯 6ch S16 buffer → 每幀取第 0 個 16-bit 樣本（ch0）；不完整尾幀丟棄。
- `_ArecordSource.read(n)`：自 stdout 讀 `n*6` bytes → `_extract_ch0` → 回傳 mono ch0（≤ n bytes）。其餘管線（send_loop / ws / Deepgram 參數）不變。

| 項目 | 規約 |
|---|---|
| 聲道假設 | 裝置 6ch（ReSpeaker 6-channel 韌體）；ch0 = 處理後 ASR 軌 |
| 不足一幀的尾 bytes | 丟棄（下次 read 自然續上；streaming 無感） |
| EOF / 半幀 | `_extract_ch0` 對 < 1 幀回 `b""`；send_loop 既有 `if not chunk: break` 處理 |

### 增量 2 — 預熱連線（gate）
- `SttWorker` 加 `_live`（`threading.Event`）：receiver 只在 `_live` set 時 print + 注入 sink；未 set 則靜默丟棄。
- 抽出 `_start_session(live)`（現 `arm()` 起 session 的本體；**起 receiver/sender thread 前依 `live` 先定閘**——避免 receiver 早於設閘而誤丟首筆，race-safe）。
- `prewarm()`：`_start_session(live=False)`（起 session、閘關、丟棄期）。
- `arm()`：無 session → `_start_session(live=True)`；已 prewarm 的 session → 直接 `_live.set()` 開閘。**單獨 arm 仍等同舊行為**（起 session 並注入）→ 既有測試/流程不破。
- `disarm()`：`_live.clear()` + 拆 session（既有邏輯不變）。
- `main.py read_customer_input`：開頭 `stt.prewarm()`（疊 TTS 播放）→ `tts.wait_idle()` → `stt.arm()` → `finally: stt.disarm()`。

| 場景 | 行為 |
|---|---|
| 預熱期（TTS 播放中）收到 speech_final | 丟棄（機器人自聲）、不 print、不注入 |
| go-live 後收到 speech_final | print + 注入（既有正規化路徑不變） |
| 缺 key / 連線失敗 | 同既有：停用 / 放棄本輪；prewarm 與 arm 皆 no-op、不炸（純鍵盤照常） |
| disarm | `_live` 歸零 + 拆 session；冪等 |
| ws keepalive | 預熱期 sender 照送音訊（機器人聲）→ ws 不因閒置 timeout；該音訊被 Deepgram 轉錄後在 receiver 端丟棄（額度成本可忽略） |
