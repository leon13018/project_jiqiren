# perf_w2 — TTS prefetch＋常駐 event loop（spec — WHAT）

> perf campaign Phase 3 最終波（唯一有體感的波）。傘狀：`resources/specs/perf_review_2026-06-12_design.md`；
> findings：F-4＋F-5（P1 hot）＋F-14（隨本波驗證點）。
> **多線程高風險波**：Windows 只能驗 fake 注入邏輯，真實效果靠 Pi 實測（pineedtodo）。

## 1. 背景與動機

現行 `TtsWorker._process`：synth（edge-tts 雲端，Pi 上 ~0.5–2s）→ play（mpg123）→ drain，
嚴格序列化。**連發句**（hawk→L2 entry、L4 循環刷新）之間有一整段「靜默等下一句合成」。

- **F-5**：`asyncio.run()` 每句重建＋銷毀 event loop——改 worker thread 常駐 loop
  （亦是 F-4 的基礎設施）。
- **F-4**：當前句播放期間（`Popen` 後、`wait()` 前）預先合成 queue 裡下一句到
  **另一個 buffer 檔**——句間靜默從「一次 synth round-trip」降到接近 0。
  單線程內重疊（不開新 thread）：等播放的時間本來就是閒置的。

## 2. 設計核心與行為規約

### 2-0. 不變式（reviewer 逐條核）

1. FIFO 順序不變（單 queue 單消費者；prefetch 只「偷看」不取出）。
2. `_pending` 語意不變：say 原子 inc、on_done dec——prefetch 不碰計數，
   `wait_idle`／`speak_and_wait` 的 wall-clock budget 起算點（最後一句真播完）不變。
3. print 在 caller thread、stdin=DEVNULL、ALSA drain 0.3s、失敗 noisy print 字面、
   `wait_idle` default 30s、shutdown 順序（terminate→drain）全部不動。
4. 基底 `QueueWorker` 零改動（無 except-all、shutdown 不入基底等既定決策維持）。
5. 效能非行為：句間靜默縮短是本波目的；synth 失敗的句子可能被嘗試兩次
   （prefetch 失敗→自己的 `_process` 內重試），最終失敗訊息字面不變、只印一次。

### 2-1. F-5：常駐 event loop

```python
    def on_thread_start(self) -> None:
        # perf_w2 F-5：worker thread 常駐 event loop（取代每句 asyncio.run 建拆）。
        # 顯式 new_event_loop（worker thread 無預設 loop，get_event_loop 會炸——
        # threading reference Part A 地雷區）；set_event_loop 讓 edge-tts 內部
        # get_event_loop 類呼叫也拿到同一顆。
        # 不在 shutdown close：daemon thread 與 loop 同壽命，main.py 以 os._exit(0)
        # 強退（S6 教訓 3/4），跨 thread close 反引入 race。
        self._loop_obj = asyncio.new_event_loop()
        asyncio.set_event_loop(self._loop_obj)
```

`_process` 階段 1 的 `asyncio.run(...)` → `self._loop_obj.run_until_complete(...)`。

### 2-2. F-4：1-deep prefetch（雙 buffer）

**資料結構**（`__init__` 於 `super().__init__()` 前設，依基底警語）：

```python
        self._buf_idx = 0            # 雙 buffer 輪替指標（僅 worker thread 觸碰）
        self._prefetched = None      # (text, mp3_path) | None（僅 worker thread 觸碰）
```

模組常數：`TMP_MP3 = "/tmp/last_tts.mp3"` 改為

```python
# 雙 buffer（perf_w2 F-4）：當前句播放中預合成下一句到另一檔，避免覆寫播放中的檔
_TTS_BUF_PATHS: tuple = ("/tmp/last_tts_0.mp3", "/tmp/last_tts_1.mp3")
```

**兩個 worker-thread-only helper**：

```python
    def _next_buf(self) -> str:
        """輪替取下一個 buffer 路徑（每次 synth 前呼叫 → 連續兩次 synth 必異檔）。"""
        self._buf_idx ^= 1
        return _TTS_BUF_PATHS[self._buf_idx]

    def _peek_next(self) -> "str | None":
        """偷看 queue 下一筆（不取出）。安全性：本 thread 是唯一消費者（唯一會移除
        元素的人），producer 並發 put 只 append 尾端 → mutex 下讀 queue[0] 穩定。"""
        with self._q.mutex:
            return self._q.queue[0] if self._q.queue else None
```

**`_process` 改造**（控制流；錯誤分支字面全部不變）：

```
階段 1（synth，改）：
    if self._prefetched is not None and self._prefetched[0] == text:
        mp3_path = self._prefetched[1]; self._prefetched = None   # prefetch 命中，跳過 synth
    else:
        self._prefetched = None                                    # 防禦：FIFO 單消費者下不可能 mismatch，丟棄即可
        mp3_path = self._next_buf()
        try: self._loop_obj.run_until_complete(_synthesize(text, mp3_path))
        except Exception: ...（noisy print 字面不變）; return

階段 2（play，僅路徑變數化）：
    Popen(["mpg123", "-q", mp3_path], stdin=DEVNULL)（CalledProcessError 的 cmd 同步用 mp3_path）

階段 2.5（prefetch，新；spawn 後、wait() 前）：
    nxt = self._peek_next()
    if nxt is not None:
        buf = self._next_buf()
        try:
            self._loop_obj.run_until_complete(_synthesize(nxt, buf))
            self._prefetched = (nxt, buf)
        except Exception:
            self._prefetched = None
            # 預取失敗刻意靜默：該句輪到自己的 _process 會重試 synth，
            # 屆時才走既有 noisy 失敗 path——避免同一句印兩次失敗訊息

階段 3（wait → returncode 檢查 → drain）：不變
```

**正確性論證**（spec 留檔，reviewer 核）：

- **buffer 不互踩**：`_next_buf` 每次 synth 前輪替 → 連續兩次 synth 必寫異檔；
  播放中的檔是「上一次 synth」的檔 → prefetch 寫的是另一個。
- **重疊即省時**：`_process(N)` 耗時從 `synth(N)+play(N)+drain` 變
  `synth(N，命中則 0)+max(play(N), synth(N+1))+drain`——synth 藏進播放等待。
- **shutdown 互動**：terminate 殺 mpg123 → 本 thread 若正卡在 prefetch synth，
  完成後 `wait()` 回 -15 走既有 CalledProcessError path；queue 已被 drain →
  prefetch buffer 成無害孤兒暫存檔。與現行 shutdown 時序同形。
- **單 thread**：`_buf_idx`／`_prefetched`／`_loop_obj` 只被 worker thread 讀寫，
  零新增鎖、零新 race 面。

### 2-3. 測試（TDD，新增 3 個於 `test_tts_worker.py`，sales 套件 → 512）

1. `test_prefetch_synthesizes_next_during_playback`：gated FakePopen 卡住第一句播放；
   斷言第二句在播放釋放**前**已被 synth（Event 同步），且全程結束後第二句只 synth 一次
   （prefetch 命中不重合成）。
2. `test_prefetch_failure_falls_back_to_inline_synth`：fake synth 對第二句首呼 raise、
   次呼成功；斷言該句最終成功播出（wait_idle True）且 synth 被呼叫兩次（prefetch 失敗
   →inline 重試），整體 `_pending` 歸 0。
3. `test_consecutive_synth_paths_alternate_buffers`：recording fake synth 收 out_path；
   連發 3 句後斷言每句恰 synth 一次且**相鄰兩次 synth 的 out_path 必不同**
   （`_next_buf` 輪替性，與 prefetch 是否命中無關的確定性質）。

既有 7 個 tts 測試零修改（fake 注入點 `_synthesize`／`Popen`／`time.sleep` 不變）。
