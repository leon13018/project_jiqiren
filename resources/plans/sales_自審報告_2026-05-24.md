## A 類 — 需立即決定

### A1. `states.py` 開頭 docstring 過時（事實錯誤，立即修）

**現況：** `states.py` line 1-12 docstring 寫：
```
"""L0-L1 各層鏈路實作（S1 v2）。
對應規格書：
    - L0：共通子例程 A（「回 L1 叫賣」）
    - L1：商家模式選擇層（叫賣 / 待機 / 客服）
    - L2-L5：尚未實作（TODO）

設計原則：
    - 對外動作以 callback 注入（speak / mute_opencv / unmute_opencv / schedule）
    ...
"""
```

**問題：**
- 標題「L0-L1」漏 L2/L3/L4/L5
- 寫「L2-L5：尚未實作（TODO）」事實錯誤
- callback 列表「speak / mute_opencv / unmute_opencv / schedule」只反映 L0-SUB-A，沒含 L1-L5 用的 `print_terminal` / `read_terminal_key` / `read_customer_input` / `opencv_*` / `exit_program` / `sleep` 等
- 每輪 subagent 沒更新

**問題根源：** 之前 L1-L5 subagent prompt 沒明確要求每輪更新檔案開頭 docstring（subagent surgical 只加新 function）。

**建議：** 立即補正（純文件改動，1 個 commit）。

**決策：** ☐ 立即修 ☐ 不必修

### A2. `logic.py` 完全空殼 — 5 層串接邏輯放哪？

**現況：** `logic.py` 只有 docstring + `# TODO: S1 v2 實作（待 BDD/TDD 規劃完成後動手）`。

**問題：** L0-L5 全寫完了，但 **沒有任何 caller 把 5 層串起來**。L1 → L2 → L3 → L4 → L5 → 子例程 A → L1 這個 cycle 沒有實作。沒有頂層持有 cart / think_count / loop_count / unclear_count 的程式碼。

**選項：**
- **A2-a：** logic.py 寫 `run()` 主迴圈 dispatcher，含全 5 層 cycle + state 持有
- **A2-b：** logic.py 仍空殼，所有串接邏輯寫進 `myProgram.py` 入口層
- **A2-c：** logic.py 寫純 dispatch 邏輯（state machine），callback wire-up 寫 `myProgram.py`

**架構文件 `backend-module-structure.md` 寫的：** `logic.py` 是「主迴圈 + 5 層 dispatch；唯一允許持有『外部世界』的進入點」。所以原本設計是 A2-a。

**疑點：** 但「外部世界」最終 wire-up（terminal input / TTS 播放 / OpenCV / 廠商 SDK）必須是 `myProgram.py` 做，因為 sales/ 嚴格不 import 廠商 SDK（選項 C）。所以變成 A2-c？

**決策：** ☐ A2-a ☐ A2-b ☐ A2-c ☐ 其他

### A3. `run_l?` callback 集合分散（8 種 variant）— 入口層 wire-up 怎麼設計？

**現況：** 5 層 + 子例程 A 共 6 個入口函式，callback 集合完全不一致：

| 函式 | callback 集合 |
|---|---|
| `run_subroutine_a` | speak / mute_opencv / unmute_opencv / schedule |
| `run_l1` | print_terminal / read_terminal_key / opencv_dwell_seconds / opencv_disable / opencv_enable / speak / exit_program / schedule |
| `run_l2` | speak / do_action / print_terminal / read_customer_input / cart / think_count |
| `run_l3` | speak / do_action / print_terminal / read_customer_input / cart / think_count |
| `run_l4` | speak / do_action / print_terminal / read_customer_input / cart / loop_count / unclear_count |
| `run_l5` | speak / do_action / mute_opencv / cart / sleep |

**問題：**
- 入口層 wire-up 時要管理 8+ 種 callback 名稱
- 每層的 callback 集合是子集，但沒有統一介面
- `read_terminal_key` (L1) vs `read_customer_input(timeout)` (L2-L4) 是兩個不同 callback — L1 用商家終端鍵盤，L2-L4 用顧客輸入（語音或終端）
- 時間排程 3 種模型：`schedule(seconds, fn)` (子例程 A, L1) / `read_customer_input(timeout)` (L2-L4) / `sleep(seconds)` (L5)
- `mute_opencv` 只 L0/L5 用；`opencv_disable/enable/dwell_seconds` 只 L1 用

**選項：**
- **A3-a：** 維持現況分散，入口層自己管理 8 種 callback（function-style wire-up）
- **A3-b：** 設計統一的 `Context` dataclass / class 把所有 callback 包進去（class-style wire-up，每層接 ctx）
- **A3-c：** 介於中間 — 分組為 `IOContext`（語音 / 動作 / 終端 / OpenCV）+ `TimeContext`（schedule / sleep / read with timeout）
- **A3-d：** 延後決定，先按現況跑 `myProgram.py` 驗證可行性，發現太亂再 refactor

**主 agent 傾向：** A3-d → 先讓入口層直接 wire 通驗證能跑，再看實際痛點決定要不要包 Context。

**決策：** ☐ A3-a ☐ A3-b ☐ A3-c ☐ A3-d

### A4. L4-A → L5 cart 清空時機保證

**現況：**
- L4-A 掃碼成功 `return ("L5", 0, 0)` — **不清空 cart**
- L5 進入時 `cart_module.clear_cart(cart)` 清空

**規格設計：** L4-A 與 L5 是必定串接（L4-A 唯一出口就是 L5），cart 在 L5 進入時清意圖是「保留 L5 期間可引用 cart 的擴充空間」（如未來在 L5 印明細）。

**疑點：** 如果入口層 caller 沒立刻呼叫 `run_l5(cart=...)`（例：邊緣情境程式被打斷、bug、refactor），cart 永遠不會清，下次進 L2 時 cart 仍含上次商品。

**選項：**
- **A4-a：** 維持現況 — 信任 L4-A 必呼 L5（規格保證）
- **A4-b：** L4-A 與 L5 共用同一個清空（雙重保險）— L4-A 也清 + L5 仍清
- **A4-c：** 入口層 wire-up 加 invariant check（每進新層前確認 cart 狀態符合預期）
- **A4-d：** `run_l4` return `("L5", ...)` 改成直接呼叫 `run_l5` chain（無 caller 中介）— 但破壞當前 return tuple 模式

**主 agent 傾向：** A4-a + A4-c — 入口層 invariant check 是 defensive 但便宜

**決策：** ☐ A4-a ☐ A4-b ☐ A4-c ☐ A4-d ☐ 其他

## B 類 — 可延後決定（影響可讀性 / refactor，不影響功能）

### B1. `run_l?` return shape 不一致（3 種）

| 函式 | return shape |
|---|---|
| `run_l1` | `str \| None`（"L2" 或 None） |
| `run_l2` / `run_l3` | `tuple[str, int]`（next_state, think_count） |
| `run_l4` / `run_l5` | `tuple[str, int, int]`（next_state, loop_count, unclear_count） |

入口層 dispatch 需要分 3 種處理。可考慮統一成 dataclass `Transition(next_state, ctx_update_dict)` 或最簡 3-tuple shape with 0 填補。

---

### B2. time 模型 3 種並存

| 模型 | 用於 |
|---|---|
| `schedule(seconds, callback)` | 子例程 A / L1 叫賣輪播（fire-and-forget 排程） |
| `read_customer_input(timeout) -> str \| None` | L2 / L3 / L4（同步等顧客回應） |
| `sleep(seconds)` | L5（純等待） |

實際上 L5 的 `sleep` 等於 `read_customer_input(timeout=THANK_DELAY)` 忽略結果。可統一改用 `read_customer_input` 但語義略奇怪（L5 規格寫「不接受任何顧客輸入」）。

---

### B3. 常數命名不一致

進入時語音常數命名不統一：
- L2: `L2_GREETING_PROMPT`
- L3: `L3_FOLLOWUP_PROMPT`
- L4: `L4_ENTRY_VOICE_TEMPLATE`（f-string 模板）
- L1: `L1_HAWK_ENTER_PROMPT` / `L1_STANDBY_ENTER_PROMPT`

建議：統一為 `L?_ENTRY_PROMPT` / `L?_ENTRY_VOICE` 或 `L?_<chain>_<purpose>` pattern。但純風格，不影響功能。

---

**現況：** `states.py` 約 1000+ 行（L0 子例程 A + L1 + L2 + L3 + L4 + L5）。

**架構文件擴展觸發條件：** 「states.py >300 行 → 拆 states/l1.py ~ l5.py」

**已超過觸發條件 3 倍以上**。建議：拆。

**選項：**
- **B4-a：** 立即拆（屬 incremental rebuild，需重跑全 pytest 確認不破）
- **B4-b：** 延後，等 L 層需要再修時順便拆（YAGNI）
- **B4-c：** 不拆，因為 sales/ 全寫完了 prod code 進入穩定期，拆只增加 import 成本

**主 agent 傾向：** B4-b — 後續 incremental rebuild S2-S7 階段勢必要動 states.py（為了非阻塞 IO），到時順便拆
