# L3：詢問額外需求（顧客互動 — 加單迴圈）

> **角色：顧客。** 已從 L2 加了第一件商品，這裡是「還要不要再加」的迴圈。**唯一進 L4 結帳的入口層**。

---

## 入口（從何處進來）

| 來源 | 條件 |
|---|---|
| L2 | 鏈路 C（點到商品 + 加單）完成 |
| L3 客服模式返回 | 鏈路 B-2 印完電話後 |
| L3 加單後繼續 | 鏈路 B-3 點到商品加單後 |

---

## 進入時動作

- **[語音]** `"請問還有額外需要購買的嗎？"`
- **[動作]** TBD（S3 加動作時補；建議點頭）
- 等待顧客回應，最多 `WAIT_NO_RESPONSE` 秒（= 6s）

> **與 L2 的時序差異（重要）**：L3 的 6s 超時**不是「拒絕」**，而是觸發「即將自動結帳」flow（見鏈路 C-2）。L2 的 6s 超時則直接走拒絕。

## 邏輯鏈路

### 鏈路 A — 拒絕

- **trigger**：顧客回應命中「拒絕意圖」關鍵字（見 [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單)）
  - ⚠️ **L3 拒絕不含 6s 超時**（6s 超時走鏈路 C-2）
- **進入動作**：
  - [語音] TBD（S2 補；建議 `"好的，取消這次購物，謝謝光臨"`）
  - **清空 `cart`**（顧客明確拒絕，已加的商品也丟掉）
- **轉到**：L1 叫賣，**套用** [L0 子例程 A：回 L1 叫賣](L0_共通.md#子例程-a回-l1-叫賣)

> **設計推論（待 review）**：「拒絕」 = 整單作廢。若你的原意是「拒絕只是不再加單、但前面加的還是要結」，請告訴我，我改成不清空 cart 直接進 L4。

### 鏈路 B — 循環詢問額外需求

此鏈路有三種子情況：

#### B-1：無法判斷

- **trigger**：顧客回應**不命中** [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單) 任一類別
- **進入動作**：
  - **[語音]** `"抱歉，我無法判斷您的需求。請問還需要購買什麼東西嗎？或者是聯繫客服。"`
- **保持在本層**：回到「等待顧客回應（最多 `WAIT_NO_RESPONSE` 秒）」

#### B-2：想找客服

- **trigger**：顧客回應命中「想找客服」關鍵字
- **進入動作**：[終端] 印 [L0 客服電話](L0_共通.md#-客服電話placeholder)
- **轉到**：印完**自動回 L3 循環狀態**（自動返回語義）

> **客服模式從 L3 進入的返回語義**：跟 L1 / L2 客服一樣**自動返回**。L3 雖然有「進行中的 cart」，但客服只是「印電話」這一個一次性動作、立即結束 — 不需要使用者手動退出。對比 L4 客服（需手動「退出 / 繼續」）。

#### B-3：點到商品（額外加單）

- **trigger**：顧客回應命中「商品 — 冰紅茶」OR「商品 — 刮刮樂」關鍵字
- **進入動作**：
  - **解析數量**：依 [L0 數量解析](L0_共通.md#-數量解析) 從顧客輸入抽取數量
  - 將命中商品 + 數量**加入既有 `cart`**（同商品累加數量 / 或 append 新條目，看實作）
  - **[語音]** `"請問還需要什麼東西嗎？"`（比 L2 / L3 進入動作簡短）
- **保持在本層**：回到「等待顧客回應（最多 `WAIT_NO_RESPONSE` 秒）」

#### B-4：想一下（顧客猶豫）

- **trigger**：顧客回應命中「想一下意圖」關鍵字（`等等` / `等一下` / `稍等` / `想想` / `考慮` / `想一下` / `hold on` / `wait`，見 [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單)）
- **進入動作**：
  - 機器人**沉默等待** `WAIT_NO_RESPONSE`（= 6s）給顧客思考時間
  - **分流**：
    - 6s 內顧客**有任何回應** → 立即處理該回應（重跑 [判定優先序](#判定優先序)）
    - 6s **仍無回應** → **[語音]** `"請問還需要什麼東西嗎？"` → 回到 L3 主等待循環（鏈路 B-3 結束後一樣的狀態）

> **與 C-2 「兩段 6s 自動結帳」的差異**：
> - **B-4「想一下」**：顧客**主動說了**「等等」這類詞 → 給 6s 思考 → 6s 後**繼續問還要什麼**（保持在 L3、不結帳）
> - **C-2「自動結帳」**：顧客**完全沒回應**滿 6s → 提示「6 秒後自動結帳」→ 再等 6s → 進 L4
> 兩者起點不同（一個有命中關鍵字、一個完全靜默）、終點不同（一個留 L3、一個進 L4）。

### 鏈路 C — 準備進 L4（結帳）

此鏈路有兩種子情況：

#### C-1：顧客主動表達結帳意圖

- **trigger**：顧客回應命中「結帳意圖」關鍵字（`結帳` / `買單` / `付款` / `好了` / `就這樣` / `可以了` 等，見 [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單)）
- **進入動作**：[語音] TBD（S2 補；建議 `"好的，為您結帳"`）
- **轉到**：**L4**

> **釐清點 #2 已敲定（2026-05-24）**：原 L3.md 寫「客戶說『等等』類似話語進 L4」 — 經使用者澄清，「等等」**不是進結帳**，而是「顧客還想想要買什麼」 → 改走新的鏈路 B-4「想一下」（保持在 L3、給 6s 思考、6s 後繼續問）。鏈路 C-1 的「結帳意圖」白名單因此**不含「等等」**。

#### C-2：超時自動進結帳（兩段 6s）

- **trigger**：等待 `WAIT_NO_RESPONSE`（= 6s）內無任何回應
- **進入動作（第一段）**：
  - **[語音]** `"因為您沒有做任何回覆，我將在 6 秒後自動進入結帳模式"`
  - 再等待 `L4_AUTO_CHECKOUT_NOTICE`（= 6s）秒
- **第二段分流**：
  - **第二段 6s 內有任何回應** → 取消自動結帳，**回到鏈路 B 處理**該回應（B-1 / B-2 / B-3 依關鍵字判定）。此時若回應命中拒絕關鍵字 → **走鏈路 A 拒絕** / 若命中結帳意圖 → 走鏈路 C-1
  - **第二段 6s 仍無回應** → 直接進 **L4**

## 判定優先序

依 [L0 關鍵字白名單判定邏輯](L0_共通.md#-關鍵字白名單)，L3 **使用全部 6 步**（與 L2 不同 — L3 接受「想一下」+「結帳意圖」）：

1. 第一段 6s 超時 → 鏈路 C-2 第一段（提示「6 秒後自動結帳」）
2. 拒絕意圖 → 鏈路 A
3. **想一下意圖** → **鏈路 B-4**（保持在 L3、給 6s 思考）
4. 結帳意圖 → 鏈路 C-1
5. 想找客服 → 鏈路 B-2
6. 商品命中 → 鏈路 B-3（加單繼續循環）
7. 都沒命中 → 鏈路 B-1（無法判斷）

> 鏈路 C-2 第二段觸發後（顧客有回應），或鏈路 B-4 思考期內顧客有回應 — 該回應**重跑 1-7 順序**。
