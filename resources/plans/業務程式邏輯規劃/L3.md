# L3：詢問額外需求（顧客互動 — 加單迴圈）

> **角色：顧客。** 已從 L2 加了第一件商品，這裡是「還要不要再加」的迴圈。**唯一進 L4 結帳的入口層**。

---

## 入口（從何處進來）

| 來源 | 條件 |
|---|---|
| L2 | 鏈路 C（點到商品 + 加單）完成 |
| L3 客服模式 B-2 | 印完電話自動返回此循環 |
| L3 鏈路 B-1 無法判斷 / B-3 點到商品加單後繼續 / B-4 想一下 6s 後重問 | 自動回到等待狀態 |
| L3 鏈路 C-2 第二段內顧客有回應重判 | 自動回到等待狀態 |

## 進入時動作

- **[語音]** `"請問還有額外需要購買的嗎？"`
- **[動作]** TBD（S3 加動作時補；建議點頭）

## 邏輯鏈路

### 鏈路 A — 拒絕

- **trigger**：顧客回應命中 **L3 嚴格 reject 詞**：`我不要了` / `不想買了` / `取消購買` / `退出` / `不買了`
  - ⚠️ **L3 拒絕不含 6s 超時**（6s 超時走鏈路 C-2）
  - ⚠️ **2026-05-25 修訂**：L3 不再用 L0 一般拒絕詞（「不要」/「不用」/「不想」/「不買」）觸發拒絕 — L3 視這些短詞為「**不需要加購**」→ **走鏈路 C-1 結帳**（同 `no`/`nope` 在 L3 的處理）。只有明確完整意圖（如「我不要了」、「取消購買」）才算拒絕。L2/L4 仍用 L0 一般 reject 詞觸發拒絕（無影響）。
- **進入動作**：
  - [語音] TBD（S2 補；建議 `"好的，取消這次購物，謝謝光臨"`）
  - **清空 `cart`**（顧客明確拒絕，已加的商品也丟掉）
- **轉到**：L1 叫賣，**套用** [L0 子例程 A：回 L1 叫賣](L0_共通.md#子例程-a回-l1-叫賣)

> **拒絕語義（已敲定）**：拒絕 = 整單作廢，清空 `cart`。

### 鏈路 B — 循環詢問額外需求

此鏈路有三種子情況：

#### B-1：無法判斷

- **trigger**：顧客回應**不命中** [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單) 任一類別
- **進入動作**：
  - `unclear_count += 1`
  - **若 `unclear_count >= UNCLEAR_MAX`（= 3，2026-05-25 加）**：進「最終確認」子狀態（仿 L4 D）：
    - **[語音] + [終端]** `L3_UNCLEAR_FINAL_PROMPT = "系統將取消這次購物，請選擇『取消訂單』或『繼續加單』（語音說或終端輸入 1=取消 / 2=繼續 皆可）"`
    - 給 `WAIT_NO_RESPONSE`（= 6s）wall-clock 倒數
    - **顧客回應分流：**
      - 終端 `1` / 語音「退出」/「取消」/「不要了」等（用 `l4_service` mode 分類）→ 走鏈路 A 拒絕（清空 cart 退 L1）
      - 終端 `2` / 語音「繼續」/「continue」等 → reset `unclear_count = 0` + 重播 `L3_ENTRY_PROMPT` + 回主等待迴圈
      - 6s 內無回應 → 視為取消，走鏈路 A
      - 亂回答 → 重印 prompt，**6s 倒數不重置**（time.time() wall-clock 持續累積）
  - **否則（`unclear_count < UNCLEAR_MAX`）**：
    - **[語音]** `"不好意思我聽不太懂，請問還要買什麼呢？或者您想聯繫客服？"`
    - 保持在本層：回到「等待顧客回應（最多 `WAIT_NO_RESPONSE` 秒）」
- **`unclear_count` 重置時機**：任何被識別意圖（拒絕 / 想一下 / 結帳 / 客服 / 商品）觸發時重置為 0；最終確認子狀態選「繼續」後也 reset

#### B-2：想找客服

- **trigger**：顧客回應命中「想找客服」關鍵字
- **進入動作**：[終端] 印 [L0 客服電話](L0_共通.md#-客服電話placeholder)
- **轉到**：印完**自動回 L3 循環狀態**（自動返回語義）

> **客服模式從 L3 進入的返回語義**：跟 L1 / L2 客服一樣**自動返回**。L3 雖然有「進行中的 cart」，但客服只是「印電話」這一個一次性動作、立即結束 — 不需要使用者手動退出。對比 L4 客服（需手動「退出 / 繼續」）。

#### B-3：點到商品（額外加單）

- **trigger**：顧客回應**至少含一個商品 keyword**（走 `parse_products`，2026-05-25 從單商品升級為多商品 B 方案；與 L2 鏈路 C 邏輯一致）
- **進入動作**：
  - **多商品解析 + 各自追問**：跟 L2 鏈路 C 同流程（見 L2.md 鏈路 C），重複商品累加到既有 `cart`
  - 範例（cart 已有冰紅茶 ×1 從 L2 帶來）：
    * 顧客「再來紅茶 1 刮刮樂 2」→ cart = {冰紅茶: 2, 刮刮樂: 2}
    * 顧客「再 3 瓶紅茶」→ cart = {冰紅茶: 4}（同商品累加）
  - **[語音]** `"請問還需要什麼東西嗎？"`（重問 L3_REASK）
- **保持在本層**：回到「等待顧客回應（最多 `WAIT_NO_RESPONSE` 秒）」

#### B-4：想一下（顧客猶豫）

- **trigger**：顧客回應命中「想一下意圖」關鍵字（`等等` / `等一下` / `稍等` / `想想` / `考慮` / `想一下` / `hold on` / `wait`，見 [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單)）
- **進入動作**：
  - `think_count += 1`
  - **若 `think_count == 3`（第 3 次說「等等」）**：
    - 跳過 6s 沉默
    - **[語音]** `f"請問是否要結帳？如果沒回應，{AUTO_CHECKOUT_NOTICE} 秒後將為您結帳"`（與鏈路 C-2 第一段同語音）
    - **直接走鏈路 C-2 第二段邏輯**（等待 `AUTO_CHECKOUT_NOTICE`=10s）
  - **否則（`think_count < 3`）**：
    - 機器人**沉默等待** `WAIT_NO_RESPONSE`（= 6s）給顧客思考時間
    - **分流**：
      - 6s 內顧客**有任何回應** → 立即處理該回應（重跑 [判定優先序](#判定優先序)）
      - 6s **仍無回應** → **[語音]** `"請問還需要什麼東西嗎？"` → 回到 L3 主等待循環
- **`think_count` 重置時機**：鏈路 B-3（加單成功）/ 鏈路 A（拒絕）/ 鏈路 C-1（結帳意圖）觸發時重置為 0

> **與 C-2 「兩段自動結帳」的差異**：
> - **B-4「想一下」**：顧客**主動說了**「等等」這類詞 → 給 6s 思考 → 6s 後**繼續問還要什麼**（保持在 L3、不結帳）；第 3 次觸發時直接進 C-2 第二段邏輯
> - **C-2「自動結帳」**：顧客**完全沒回應**滿 6s → 提示「10 秒後自動結帳」→ 再等 10s → 進 L4
> 兩者起點不同（一個有命中關鍵字、一個完全靜默）、終點不同（一個留 L3、一個進 L4）。

### 鏈路 C — 準備進 L4（結帳）

此鏈路有兩種子情況：

#### C-1：顧客主動表達結帳意圖

- **trigger**：顧客回應命中「結帳意圖」關鍵字（`結帳` / `買單` / `付款` / `好了` / `就這樣` / `可以了` 等，見 [L0 關鍵字白名單](L0_共通.md#-關鍵字白名單)）
- **進入動作**（2026-05-25 加 confirm 子狀態，B 方案）：
  1. 進「結帳確認」子狀態：
     - **[終端 + 語音]** `L3_CHECKOUT_CONFIRM_TEMPLATE`：「您即將結帳，總共 {summary}，金額 {total} 元，正確嗎？（語音說『對』/『不對』，或終端輸入 1=對 / 2=不對）」
     - `WAIT_NO_RESPONSE`（= 6s）wall-clock 倒數
     - **顧客回應分流：**
       * 終端 `1` / 語音「對」/「是」/「好」/「確認」等（`KEYWORDS_CONFIRM_YES`）→ 進 L4
       * 終端 `2` / 語音「不對」/「錯」/「改」等（`KEYWORDS_CONFIRM_NO`）→ caller speak `L3_CHECKOUT_CONFIRM_REJECT_VOICE`「好的，請告訴我您要的東西」+ 重播 `L3_ENTRY_PROMPT` 回 L3 主等待迴圈（cart 保留）
       * 6s 內無回應 → 視為確認（顧客已主動說結帳，沒拒絕就過）→ 進 L4
       * 亂回答 → 重印 prompt，**6s 倒數不重置**（time.time() wall-clock）
  2. 確認通過 → [語音] `L3_C1_CHECKOUT_GO = "好的，為您結帳"` → 進 L4
- **轉到**：**L4**（確認後）/ **L3 主等待**（顧客否認後）

> **Why confirm**：多商品點單（B 方案）+ 重複 utterance 累加可能造成誤增。進結帳前再 confirm 一次給顧客機會修正。C-2（沈默自動結帳）已有第一段「請問是否要結帳？10s 後將結帳」性質的 confirm，故不重複加。

> **釐清點 #2 已敲定（2026-05-24）**：原 L3.md 寫「客戶說『等等』類似話語進 L4」 — 經使用者澄清，「等等」**不是進結帳**，而是「顧客還想想要買什麼」 → 改走新的鏈路 B-4「想一下」（保持在 L3、給 6s 思考、6s 後繼續問）。鏈路 C-1 的「結帳意圖」白名單因此**不含「等等」**。

#### C-2：超時自動進結帳（6s + 10s 兩段）

- **trigger**：等待 `WAIT_NO_RESPONSE`（= 6s）內無任何回應
- **進入動作（第一段）**：
  - **[語音]** `f"請問是否要結帳？如果沒回應，{AUTO_CHECKOUT_NOTICE} 秒後將為您結帳"`
  - 再等待 `AUTO_CHECKOUT_NOTICE`（= 10s）秒

> 語音文案用 f-string 插值常數，未來改 `AUTO_CHECKOUT_NOTICE` 數值時文案自動同步，避免漏改。
- **第二段分流**：
  - **第二段 10s 內有任何回應** → 取消自動結帳，**回到鏈路 B 處理**該回應（B-1 / B-2 / B-3 依關鍵字判定）。此時若回應命中拒絕關鍵字 → **走鏈路 A 拒絕** / 若命中結帳意圖 → 走鏈路 C-1
  - **第二段 10s 仍無回應** → 直接進 **L4**

## 判定優先序

依 [L0 關鍵字白名單判定邏輯](L0_共通.md#-關鍵字白名單)，L3 **使用全部 6 步**（與 L2 不同 — L3 接受「想一下」+「結帳意圖」）：

1. 第一段 6s 超時 → 鏈路 C-2 第一段（提示「10 秒後自動結帳」）
2. 拒絕意圖 → 鏈路 A
3. **想一下意圖** → **鏈路 B-4**（保持在 L3、給 6s 思考）
4. 結帳意圖 → 鏈路 C-1
5. 想找客服 → 鏈路 B-2
6. 商品命中 → 鏈路 B-3（加單繼續循環）
7. 都沒命中 → 鏈路 B-1（無法判斷）

> 鏈路 C-2 第二段觸發後（顧客有回應），或鏈路 B-4 思考期內顧客有回應 — 該回應**重跑 1-7 順序**。

> **L3 跳過項**：~~繼續交易意圖 / 退出交易意圖~~ 在 L3 不適用（L0 已標「僅 L4 客服模式內用」）。L3 中若顧客講「繼續」「退出」「取消」「離開」等詞，視為「無法判斷」走鏈路 B-1。
