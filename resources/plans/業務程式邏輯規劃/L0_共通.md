## 🎯 層狀態判定原則（2026-05-25 B 方案敲定）

**state machine 由世界狀態（cart）驅動，非動作歷史驅動。**

層轉換條件（不論口述「在 L? 做了什麼」，以下才是定義性條件）：

| 轉換 | 觸發條件 |
|---|---|
| L1 → dialog | OpenCV 偵測到顧客（dwell ≥ `OPENCV_DWELL`）|
| **dialog 內模式切換** | **cart 空 = L2 模式（詢問需求）；cart 非空 = L3 模式（加單 / 結帳）** |
| dialog → L4 | cart 非空 + 顧客明確要求結帳（含 6s+10s 自動結帳）|
| L4 → L5 | 顧客掃碼成功 |
| 任何層 → L1（子例程 A）| 顧客拒絕 / 結帳完成 / 強制退 |

**為何 cart 狀態驅動：**
- 動作驅動 = 一旦設計「L2 加單 → 進 L3」這種轉換，未來加刪除商品時 L3 內 cart 變空仍會問「還需要什麼東西嗎？」違和。
- 狀態驅動 = dialog 每輪主迴圈讀 cart 狀態決定 prompt 與行為，刪除商品時 cart 變空，下一輪自動切回詢問需求模式（無需額外 transition）。

**對應實作（`myProgram/sales/states/dialog.py`）：**
- L2/L3 合一為單一 `run_dialog` 函式（2026-05-25 重構，原為 `run_l2` + `run_l3` 兩函式）
- 入口時 + 每輪迴圈讀 `cart_module.is_empty(cart)` 決定模式
- L2 模式行為（cart 空）：詢問需求 / timeout=A 拒絕 / 結帳意圖視為 unclear / B-3 第 3 次走 A
- L3 模式行為（cart 非空）：詢問加單 / timeout=C-2 自動結帳 / 結帳前 confirm / B-4 第 3 次進 C-2 / B-1 達 UNCLEAR_MAX 進最終確認子狀態

L2.md / L3.md 各自規格仍保留 — 視為「dialog 在某 cart 狀態下的行為描述」，不是獨立函式規格。

## ⏱ 時間常數

| 常數名 | 預設值 | 用途 |
|---|---|---|
| `WAIT_NO_RESPONSE` | `6` 秒 | L2 / L3 / L4 顧客無回應 timeout |
| `HAWK_INTERVAL` | `12` 秒 | L1 叫賣模式每一輪間隔 |
| `OPENCV_MUTE` | `12` 秒 | 「回 L1 叫賣」時屏蔽 OpenCV 偵測的時長（= `HAWK_INTERVAL`，避免折返顧客在無聲空窗期被招呼）|
| `THANK_DELAY` | `3` 秒 | L5 致謝完成後等待時長 |
| `AUTO_CHECKOUT_NOTICE` | `10` 秒 | L3 提示「自動進入結帳」前的等待時長（鏈路 C-2 第二段）|
| `L4_MAX_LOOPS` | `6` | L4 印金額循環最多次數（達到強制退）；總長 42 秒 |
| `OPENCV_DWELL` | `1.5` 秒 | OpenCV 偵測到人持續多久才視為有效觸發（防路人經過 / 商家走過 / 光影誤觸）|

> S1 v2 階段測試時可暫調短（例如 `WAIT_NO_RESPONSE=3`、`HAWK_INTERVAL=4`）加速迭代；上線前改回上述預設。

> **timeout 起算規則（已敲定 2026-05-24）**：所有 timeout 常數（`WAIT_NO_RESPONSE` / `AUTO_CHECKOUT_NOTICE` / `THANK_DELAY`）一律從**當前 TTS 播放結束時刻**起算（避免 TTS 時間吃掉顧客思考時間）。S1 v2 階段沒 TTS、用 print 模擬時，從 `print()` 結束起算。S2 接 edge-tts 時，從 TTS 播放完成回傳起算。
>
> **子鏈路返回主循環 timer 重置規則**：子鏈路（如 L2 B-3 / L3 B-4 沉默思考、L3 C-2 第二段、L4 客服模式）結束返回主循環時，主循環的 timer 從 0 重新起算，不疊加先前耗時。例：L2 B-3 沉默 6s 後重問完畢回主循環 → 主循環 `WAIT_NO_RESPONSE` 從 0 開始算，不會「立刻又因 6s 已過而觸發拒絕」。

## 🔁 共通子例程

### 子例程 A：「回 L1 叫賣」

L2 / L3 / L4 拒絕分支 / L4 第四次循環強制退 / L5 結束後 → 都套用此例程。

**步驟：**
1. 屏蔽 OpenCV 偵測 `OPENCV_MUTE`（= 12s）秒（mute 期內不偵測、不叫賣）
2. `OPENCV_MUTE` 秒後恢復 OpenCV 偵測 + **立即播放第一輪叫賣**（mute 結束無空窗期）
3. 後續每 `HAWK_INTERVAL`（= 12s）秒換下一組叫賣，依 6 組叫賣術語 mod 6 輪流使用

**範例時序：**
```
t=0    回到 L1（屏蔽 OpenCV、不叫賣）
t=12   OpenCV 恢復 + 立即叫賣（第 1 組）
t=24   叫賣（第 2 組）
t=36   叫賣（第 3 組）
t=48   叫賣（第 4 組）
...
```

`OPENCV_MUTE` 與 `HAWK_INTERVAL` 對齊（皆 12s）— OpenCV 一恢復就有叫賣聲、無「醒著但靜默」空窗期，解決顧客折返立刻被拉回 L2 的 UX 問題。

## 🛒 商品列表

| 商品 | 原價 | 折扣 | 實際價 | 單位 |
|---|---|---|---|---|
| 冰紅茶 | 30 元 | 0.9 | **27 元** | 瓶 |
| 刮刮樂 | 200 元 | 0.9 | **180 元** | 張 |

> **沿用 S1 v1 設定**（全場九折）。S1 v2 程式內以 `PRODUCTS` dict 表示，便於後續擴增 / 換品。
> **單位欄位（2026-05-25 加）**：L2 / L3 鏈路 C 商品意圖無數量時，系統用此單位追問「請問您要幾`{單位}`？」— 對冰紅茶為「瓶」，對刮刮樂為「張」。
>
> **追問 sub-loop（2026-05-25 加）**：追問內顧客回應分流（程式 `resolve_and_add_product`）：
> 1. 含數量（has_quantity）→ 用該數量加 cart
> 2. timeout（6s 無回應）→ 預設 1 加 cart（避免無限迴圈）
> 3. 客服意圖 → print 電話 + speak `QTY_CLARIFY_TEMPLATE` → 繼續追問
> 4. 拒絕意圖（L2 mode='l2' / L3 mode='normal'，依層別 reject 詞集）→ 取消加單，回主迴圈 re-prompt
> 5. 其他（想一下 / 結帳 / 商品 / 無法判斷）→ speak clarify → 繼續追問
>
> 語音常數：`QTY_PROMPT_TEMPLATE` = 初次追問；`QTY_CLARIFY_TEMPLATE` = 顧客亂說 / 客服回來時 / 其他不命中意圖時。

## 🛒 cart 生命週期

| 動作 | 時機 |
|---|---|
| 初始化（空 dict）| L2 進入時動作（若不存在則建立）|
| 第一次加入商品 | L2 鏈路 C 加單（同商品累加數量；不 append 新條目）|
| 累加 | L3 鏈路 B-3 加單（同商品累加數量；不 append 新條目）|
| 計算總額 | L4 進入時動作 |
| 清空（中斷）| L3-A 拒絕 / L4-B 拒絕 / L4-C 客服退出 / L4-D 強制退 |
| 清空（完成）| L5 進入時（L4-A 掃碼成功 → L5 後在 L5 清，保留 L5 期間可引用 cart 的擴充空間）|

> **同商品處理規則（已敲定 2026-05-24）**：相同商品名稱在 cart 內**累加數量**，不 append 新條目。例：顧客先說「冰紅茶」（cart={冰紅茶: 1}）再說「再來一杯冰紅茶」→ `cart={冰紅茶: 2}`，L4 印金額顯示「冰紅茶 ×2 = 54 元」。

## 🔢 數量解析

L2 / L3 命中商品時，要同時從顧客輸入抽取數量（例：「冰紅茶兩個」= 冰紅茶 × 2）。

**S1 v2 階段判定規則：**

| 來源 | 對應數量 |
|---|---|
| 阿拉伯數字（`re.findall(r"\d+")` 取第一個）| 直接取數值（>0 時生效）|
| 中文 `一` / `壹` | 1 |
| 中文 `兩` / `二` / `貳` | 2 |
| 中文 `三` / `參` | 3 |
| 中文 `四` / `肆` | 4 |
| 中文 `五` / `伍` | 5 |
| 中文 `六` / `陸` | 6 |
| 中文 `七` / `柒` | 7 |
| 中文 `八` / `捌` | 8 |
| 中文 `九` / `玖` | 9 |
| 中文 `十` / `拾` | 10 |
| **以上皆無命中** | **預設 1** |

**判定範例：**
- 「冰紅茶兩個」→ 商品 = 冰紅茶，數量 = 2
- 「我要刮刮樂」→ 商品 = 刮刮樂，數量 = 1（預設）
- 「3 杯冰紅茶」→ 商品 = 冰紅茶，數量 = 3
- 「給我一杯紅茶」→ 商品 = 冰紅茶，數量 = 1

> S6+ NLP 接入後可換意圖分類 + 量詞抽取，介面 `input str → (商品, qty)` 不變。

## 🔑 關鍵字白名單

S1 v2 階段「無法判斷」的判定方式：**規則匹配**（顧客輸入若不含下列任一類關鍵字 → 視為「無法判斷」）。

| 類別 | 關鍵字（任一命中即觸發） |
|---|---|
| **拒絕意圖** | `不要` / `不用` / `不想` / `不買` / `no` / `nope` |
| **想一下意圖** | `等等` / `等一下` / `稍等` / `想想` / `考慮` / `想一下` / `hold on` / `wait` |
| **結帳意圖** | `結帳` / `買單` / `付款` / `好了` / `就這樣` / `可以了` / `沒了` / `沒有了` / `夠了` |
| **繼續交易意圖**（僅 L4 客服模式內用）| `繼續` / `接著` / `繼續買` / `繼續交易` / `continue` |
| **退出交易意圖**（僅 L4 客服模式內用）| `退出` / `取消` / `離開` / `算了` / `不買了` / `exit` |
| **想找客服** | `客服` / `聯絡` / `聯繫` / `contact` / `服務` |
| **商品 — 冰紅茶** | `紅茶` / `冰紅茶` / `hong cha` / `tea` |
| **商品 — 刮刮樂** | `刮刮樂` / `刮刮` / `彩券` / `lottery` / `scratch` |

**判定邏輯（依序檢查、先命中先返回）：**

**(0) 前置：6s 內無回應 → 各層自定 timeout 行為（非關鍵字命中走向）：**
- L2 → 鏈路 A 拒絕
- L3 → 鏈路 C-2 第一段（提示「10 秒後自動結帳」）
- L4 → 鏈路 D（loop_count++，循環印金額）

**(1) 有回應時的關鍵字命中順序：**
1. 拒絕意圖 → 走拒絕分支
2. 想一下意圖（L2 / L3 皆使用）→ 走「想一下」分支（保持在本層、等 6s 後重新詢問）
3. 結帳意圖（僅 L3 進 L4 使用）→ 走「準備結帳」分支
4. 想找客服 → 走客服模式
5. 商品命中 → 走「點到商品」分支
6. 都沒命中 → **「無法判斷」**，走循環詢問分支

> **白名單詞性敲定理由（2026-05-24 終審）**：「夠了」/「沒了」/「沒有了」在實際對話中語意接近「我買完了要結帳」（顧客買到夠了 = 想結帳），不是「我不要了 = 走人」。故歸類為結帳意圖。

> **後續可換 NLP**：S6+ 之後若接 STT，可把規則匹配層換成 NLP / LLM 意圖分類，介面（input str → category）不變。S1 v2 用規則匹配即可。
