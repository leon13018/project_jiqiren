# 第二部分 — Agent B 原始報告：UX + 業務邏輯 + 待釐清點

> **視角：** 顧客體感、時間設定合理性、措辭情境、邊界情境、所有「待 review / 設計推論 / TBD」主動提選項。把規格書當「跑起來會發生什麼」逐情境推演。

## 摘要

本次審查共找出 **顧客體驗類問題 14 項**（其中 4 項嚴重度「高」、6 項「中」、4 項「低」）、**邊界情境缺漏 7 項**、**待釐清點 14 項**（含 6 項「設計推論」、4 項 TBD、4 項「隱性未明寫」）。

## 面向 1：顧客視角走 path

### 發現 B1-1：L4 鏈路 E「無法判斷」可無限循環，是嚴重 UX 陷阱
- **嚴重度**：高
- **位置**：`L4.md:104-113`
- **顧客體感**：顧客若不熟操作，可能反覆說一些非白名單的話（例如「這個多少錢？」、「我可以用信用卡嗎？」、「需要發票嗎？」）— 每講一次，機器人就回一次「抱歉，我無法判斷您的回應。您的總金額是 234 元，請掃碼，或聯繫客服」。在規格現狀下，這會永遠循環下去；而鏈路 D 的「30 秒強制退」根本永遠不會觸發（因為顧客「有」回應，不是 6s 無回應）。顧客可能在第 3 次就感到被 robot 嘲弄、第 5 次就直接走人。
- **建議修正**：必須要有 fallback。建議鏈路 E 也計入一個獨立計數器（例如 `unclear_count`），達到 3 次就強制走「請聯繫客服」+ 自動進入鏈路 C 客服模式（印電話 + 等選擇）。**詳見點 5-4 選項分析**。

### 發現 B1-2：L3 鏈路 B-4「想一下」可被顧客連環觸發、cart 永遠卡住
- **嚴重度**：高
- **位置**：`L3.md:68-80`
- **顧客體感**：顧客可一直說「等等 → (6s) → 等等 → (6s) → 等等 → ...」無限延長 L3 不進結帳。如果是 demo 場景，這倒不一定發生；但實務上某些顧客就是會反覆猶豫，或更糟 — **demo 時觀眾故意測試壓力**。
- **建議修正**：建議 B-4 也有最大重複次數（例如 2 次後，第 3 次說「等等」直接走 C-2 提示自動結帳 flow，或直接回 B-1 無法判斷 fallback）。**詳見點 5-11**。

### 發現 B1-3：L3 → L4「兩段 6s」總計 12 秒就被推進結帳，太短
- **嚴重度**：高
- **位置**：`L3.md:94-103`（鏈路 C-2）
- **顧客體感**：顧客可能只是在看商品、跟同伴小聲討論、或拿手機。第一段 6s 完全靜默 → 第二段 6s 完全靜默就強行進 L4 結帳，**12 秒**就被結帳。實務上顧客在便利店挑東西停頓 15 秒以上是很常見的，這時直接結算「冰紅茶 ×1 = 27 元」可能完全不是他真實想買的數量。
- **建議修正**：第二段時長拉長至 10-15s；或者第二段語音改成「**詢問**」而不是「**通知**」（例：`"請問是否要結帳？沒回應的話 10 秒後將自動進入結帳"`）。

### 發現 B1-4：L4「30 秒強制退」對「在掏手機 / 找 App / 連 Wi-Fi」的顧客太短
- **嚴重度**：高
- **位置**：`L4.md:94-102`
- **顧客體感**：實務上，掃碼付款流程包含：取出手機 → 解鎖 → 打開支付 App（LINE Pay / 街口 / 悠遊付）→ 連店家 Wi-Fi 或行動網路掛掉重連 → 對準 QR Code 掃描 → 確認金額 → 輸入密碼 / Face ID。**這在順利時要 15-25 秒，不順利時 40-60 秒**。30s 強制退並印「請儘快完成掃碼，否則將取消這次交易」會讓顧客覺得：「我在努力付了你還催我？」
- **建議修正**：建議 `L4_MAX_LOOPS = 6`（總長 42s）或 8（總長 54s）；或者讓「催促文案」只在 loop_count ≥ 3 才加（不是現在的 loop_count ≥ 2）。**詳見點 5-12**。

### 發現 B1-5：L5 致謝 3s 期間若 OpenCV 又抓到下一個顧客 — 規格沒處理
- **嚴重度**：中
- **位置**：`L5.md:20-34`
- **顧客體感**：L5 此時只有 3s 等待。期間下一個顧客走近：(1) OpenCV 是否正在偵測？規格沒寫；(2) 若偵測到，會走「進 L2」還是被子例程 A 的 5s mute 攔住？— 注意 mute 是「轉到 L1 後」才開始的，所以 L5 等待這 3s 內 OpenCV 沒被 mute。若同時觸發 L2，會造成致謝語音還在播 / cart 已清空但畫面切換到 L2 的詭異情境。
- **建議修正**：L5 進入時就應該屏蔽 OpenCV（與子例程 A 對齊），或者乾脆把致謝 3s 延長並列入 OPENCV_MUTE 區間（即進 L5 立刻 mute，3s 致謝 + 5s mute 合併）。**詳見點 5-9**。

### 發現 B1-6：L2 鏈路 B-1「無法判斷」3 次以上聽起來像在重複施壓
- **嚴重度**：中
- **位置**：`L2.md:38-43`
- **顧客體感**：B-1 沒有計數限制，顧客若連續講 3 次無法判斷的話（例如「呃...」、「嗯...」、「等下我想想」如果關鍵字沒命中時 — 注意「想想」其實有命中 B-3），機器人會連續念 3 次「抱歉，我無法判斷您說明的內容。您好，請問需要購買什麼東西嗎？或者您想聯繫客服。」這是一句 18 字 + 兩個分句的長語音，連念 3 次顧客會反感。
- **建議修正**：與 B1-1 對應，建議 L2/L3/L4 三層都有 `unclear_count` 上限，達標後改走「客服 fallback」。

### 發現 B1-7：B-3「想一下 6s 後語音復活」會打斷顧客思考
- **嚴重度**：中
- **位置**：`L2.md:54-62`、`L3.md:68-76`
- **顧客體感**：顧客主動說「等等」是希望機器人別吵。6s 後機器人自動說「請問需要購買什麼東西嗎？」 — 對顧客而言這就像「你跟朋友說『等等讓我想一下』，朋友 6 秒後立刻又問『所以呢？所以呢？』」。6s 是短暫思考可以、但作為「禮貌容忍時間」太短。
- **建議修正**：B-3 / B-4 的沉默時間建議拉長到 8-10s；或者第一次回問改成更柔的「您好，想好了再告訴我喔」這類非催促式語句。

### 發現 B1-8：L3 鏈路 C-2 第一段「您沒有做任何回覆」帶有指責語氣
- **嚴重度**：中
- **位置**：`L3.md:97-98`
- **顧客體感**：機器人說「**因為您沒有做任何回覆**，我將在 6 秒後自動進入結帳模式」— 「因為您沒有」這個句構在中文裡帶有「責怪、歸因於對方」的味道。顧客本來只是在看商品 / 跟同伴聊天 / 拿錢包，被機器人「點名」說「你沒回覆」會覺得被指責。
- **建議修正**：改成更中性的「我將為您結帳，若不需要結帳請說『等等』」或「準備為您結帳了，如還要加買請告訴我」。**詳見點 5-13**。

### 發現 B1-9：L2 → L1 拒絕 / 6s 超時時，沒有「再見」語音也沒有等待
- **嚴重度**：低
- **位置**：`L2.md:26-32`（鏈路 A）
- **顧客體感**：規格寫 `[語音] TBD（S2 補；建議 "謝謝光臨"）` — 但這在 S1 v2 是空的。如果 S1 v2 demo 沒語音 stub print，顧客（或評審）會覺得「機器人直接放棄了我，沒打招呼就走人了」。
- **建議修正**：S1 v2 階段也應該有 `print("[語音] 謝謝光臨")` stub，確保時序完整。

## 面向 2：時間設定合理性

### 發現 B2-1：所有 6s timeout 沒明寫「從何時起算」— 嚴重含糊
- **嚴重度**：高
- **位置**：`L0_共通.md:13`、`L2.md:18-20`、`L3.md:19-21`、`L4.md:31-33`
- **問題**：L2 進入時動作是 `[語音] "您好，請問需要購買什麼東西嗎？"`，這句 TTS 念出來大約 **2.5-3 秒**。規格寫「等待顧客回應，最多 6s」— 那 6s 是：
  - (a) 從 L2 入口開始計（含 TTS 期間）— 顧客真正能想的時間只剩 3-3.5s
  - (b) 從 TTS 結束才開始計 — 真實 6s
- 同理 L4 進入時 TTS「您的總金額是 234 元，請掃碼」也是 3-4 秒，這時 6s 從哪起算？
- **建議修正**：規格內統一明寫「`WAIT_NO_RESPONSE` timer 從 TTS 結束（或當前語音播放結束）開始計時」。S1 v2 階段沒 TTS、用 print，所以 stub 可以從 print 結束計，但 S2 加 TTS 時這個語義必須鎖死。

### 發現 B2-2：HAWK_INTERVAL=12s vs OPENCV_MUTE=5s 的時序漏洞
- **嚴重度**：中
- **位置**：`L0_共通.md:14-15`、子例程 A 範例時序 `L0_共通.md:36-43`
- **問題**：子例程 A 設計是「mute 5s → 跳過第一輪 → t=17 第二輪」。但這假設了 `HAWK_INTERVAL > OPENCV_MUTE`。如果 OpenCV 恢復偵測在 t=5，但下一輪叫賣要等到 t=17 — **t=5 到 t=17 這 12 秒空窗期**，OpenCV 是醒著的但機器人完全沒聲音；剛離開的顧客 t=8 折返就直接觸發 L2，體感上會是「我才走 8 秒、商家就立刻又招呼我」— 因為他根本沒聽到第二輪叫賣（叫賣還沒開始）。
- **建議修正**：選一個解法：(a) `OPENCV_MUTE = HAWK_INTERVAL = 12s`，mute 跟空窗期對齊；(b) 子例程 A 增加「mute 期內 OpenCV 偵測到人不觸發 L2，但記錄一個 flag，第二輪叫賣後 5s 才允許觸發」這類額外緩衝。**詳見點 5-8**。

### 發現 B2-3：THANK_DELAY=3s 對 TTS「謝謝您的光臨，歡迎再度光臨」剛好夠唸完
- **嚴重度**：低
- **位置**：`L5.md:17-20`
- **問題**：這句 TTS 唸出來約 2.5-3 秒。THANK_DELAY=3s 意思是「TTS 播完 + 等 3s」還是「總共 3s 內 TTS 播完就走」？規格沒寫。如果是後者，顧客可能還沒聽完「歡迎再度光臨」機器人就已經轉回 L1 開始叫賣下一輪，會有「未唸完就被打斷」的怪感。
- **建議修正**：明寫 THANK_DELAY 從「TTS 播完」起算。

### 發現 B2-4：L4_AUTO_CHECKOUT_NOTICE=6s 與「我將在 6 秒後」的語音耦合得太死
- **嚴重度**：低
- **位置**：`L3.md:97-98`、`L0_共通.md:17`
- **問題**：常數寫死 6s + 語音文案寫死「6 秒後」。如果未來 demo 想調成 8s，得同時改兩處。
- **建議修正**：S2 加 TTS 時，文案應動態插值：`f"因為您沒有做任何回覆，我將在 {L4_AUTO_CHECKOUT_NOTICE} 秒後自動進入結帳模式"`。S1 v2 stub 階段就先寫成動態字串避免 S2 漏改。

### 發現 B2-5：L4 整體 30s 強制退 vs 顧客結帳真實耗時不匹配
- **嚴重度**：高（與 B1-4 同源，但時間數值角度補充）
- **位置**：`L4.md:94-102`
- **數據對比**：
  | 流程 | 一般耗時 |
  |---|---|
  | LINE Pay 掃碼付款（順） | 12-20s |
  | LINE Pay（要解鎖 / 重連） | 25-40s |
  | 街口 / 全支付 / 悠遊付 | 15-30s |
  | 第一次用此 App / 找 App | 30-60s |
- 規格 30s 強制退在「順利」情境的尾端，在「不順利」就完全來不及。
- **建議修正**：L4_MAX_LOOPS 提升至 6（42s）或 8（54s）。**詳見點 5-12**。

## 面向 3：措辭情境合理性

### 發現 B3-1：L2 B-1 連念 3 次時語意累贅
- **嚴重度**：中

## 面向 5：待釐清點 — 每項給選項 + 推薦

### 點 5-1：L1 待機模式 OpenCV 是否觸發 L2（`L1.md:52`）
- **選項 A**：待機 = OpenCV 完全關閉，商家暫停就是完全暫停（規格目前推論）
- **選項 B**：待機 = OpenCV 偵測但忽略（不觸發 L2），但記錄日誌 / 計數，商家按 `r` 回來後可看到「您暫停期間有 3 位顧客經過」
- **選項 C**：待機 = OpenCV 偵測且觸發 L2（與叫賣模式一樣）
- **Trade-off**：
  - A：最單純，符合「暫停就是暫停」直覺。但商家錯失客流量數據
  - B：保留情報、不打擾，但 S1 v2 stub 沒地方顯示（要加新介面）
  - C：違反「暫停」語意，商家暫停去整理貨架，顧客一進門就被招呼，商家被打斷
- **🌟 推薦**：**選項 A**。理由：(1) S1 v2 階段最簡單；(2) 符合「暫停 = 完全暫停」直覺；(3) 選項 B 是未來增強，可在 demo 後加。

### 點 5-2：L1 叫賣模式啟動時是否套用子例程 A（`L1.md:63-64`）
- **選項 A**：不套用、第一輪不跳過（規格目前設計） — 商家按 1 立刻聽到第一組叫賣
- **選項 B**：套用、第一輪跳過 — 商家按 1 後 5s mute 才開始叫賣
- **Trade-off**：
  - A：商家有「我啟動成功了」的即時 feedback；但若商家剛開店時自己還在相機範圍內，會立刻觸發 L2 拉自己進去
  - B：避免商家觸發自己；但商家會疑惑「我按了沒反應？」
- **🌟 推薦**：**選項 A**（保持規格現狀）。理由：(1) 商家在叫賣模式啟動瞬間應該有立即反饋；(2) 商家可選擇按 1 後快速離開相機範圍。但建議加註：「叫賣模式啟動後 OpenCV mute 3s（短）」作為小防護。

### 點 5-3：L3 拒絕鏈路 = 整單作廢 vs 不再加單前面照結（`L3.md:38`）
- **選項 A**：拒絕 = 整單作廢，清空 cart（規格目前推論）
- **選項 B**：拒絕 = 不再加單，但前面加的照結 — 走到 L4 結帳
- **選項 C**：根據關鍵字細分 — 「不要了」= 整單作廢，「夠了」/「沒了」= 不再加單照結（語意更精準）
- **Trade-off**：
  - A：邏輯單純，但顧客若想表達「夠了、結帳」會被誤解為「不要了」走失整單。注意 L0 結帳意圖白名單有「好了」「就這樣」「可以了」，但 L0 拒絕白名單有「沒了」、「沒有了」、「夠了」、「不要」、「不用」— **這兩組語意在實際對話裡會混淆**（顧客說「夠了」到底是「夠了不要了」還是「夠了我要結帳」？）
  - B：對顧客友善，但語意上「拒絕」走 B 邏輯怪
  - C：最符合真實對話，但工程複雜度高、規則邊界難定
- **🌟 推薦**：**選項 A，但要把 L0 白名單調整** — 將「夠了」、「沒了」從拒絕意圖移到結帳意圖（這兩個詞語境上更接近「我買完了要結帳」而非「我不要了」）。**這是重要的白名單修正**，建議列為高優先級待釐清點。

### 點 5-4：L4 鏈路 E「無法判斷」是否該有上限（`L4.md:111`）
- **選項 A**：可無限循環無上限（規格目前設計）
- **選項 B**：設 `unclear_count` 上限 3 次，達標後自動進客服模式（鏈路 C）

  - A：邏輯最簡，但顧客可被無限折磨（已在 B1-1 詳述）
  - B：最人性化，達標後給「真人協助」出口；但顧客可能不要客服只想付款
  - C：清晰結束但對顧客冷淡
- **🌟 推薦**：**選項 B**。理由：(1) 解決 B1-1 嚴重 UX 問題；(2) 自然把不會操作的顧客導向客服取得協助；(3) `unclear_count` 在 L4 鏈路 B/A 觸發時應重置（讓計數只針對連續無法判斷）。

  - C：折衷

- **理由**：避免 S1 v2 demo 時 6 組都唸第 1 組看起來像 bug。

### 點 5-9：L5 致謝期間是否屏蔽 OpenCV（隱性未明寫）
- **選項 A**：L5 期間 OpenCV 持續偵測（規格目前隱含）
- **選項 B**：L5 進入時立即啟動 OpenCV mute（與子例程 A 對齊）— 3s 致謝 + 5s mute 串接
- **選項 C**：L5 致謝期間 OpenCV 偵測但不觸發 L2（記錄但忽略）
- **Trade-off**：
  - A：可能在 3s 內被下個顧客誤觸發（見 B1-5）
  - B：最乾淨，下個顧客至少要等 8s 才會被招呼
  - C：複雜度高、價值不大
- **🌟 推薦**：**選項 B**。理由：(1) 解決 B1-5/B4-4 問題；(2) 不影響第一個顧客體驗（他正在離開）；(3) 8s 對下一個顧客是合理的禮貌間隔。

### 點 5-10：所有層的 `[動作] TBD`（規格散見各層）
- **選項 A**：S3 統一補（規格現狀）
- **選項 B**：S1 v2 時先填 stub 動作描述，方便 S3 對應
- **🌟 推薦**：**選項 B**。建議現在就用 print 寫出方向：

  - L3 進入：`print("[動作] 點頭（傾聽姿態）")`

### 點 5-11：L3 鏈路 B-4「想一下」是否該有最大重複次數（隱性未明寫）
- **選項 A**：可無限重複（規格目前隱含） — 見 B1-2
- **選項 B**：B-4 也有 `think_count` 上限 2 次，第 3 次說「等等」直接走 C-2 自動結帳 flow

- **Trade-off**：
  - A：對真在猶豫的顧客最寬容，但 demo 風險高（觀眾故意測試）
  - B：邏輯漂亮，B-4 / C-2 對齊；但若顧客真在想，被推進結帳會錯買
  - C：較硬，但避免 cart 卡死
- **🌟 推薦**：**選項 B**。理由：(1) 與 C-2 邏輯對稱；(2) demo 時觀眾若故意測試「等等」會看到優雅 fallback；(3) 真實顧客連說 3 次「等等」其實就是不確定要不要買，推進到「您要不要結帳？」的提問是合理的。

### 點 5-13：L4 鏈路 D 催促詞觸發時機（隱性 — 語氣分級）

### 點 5-14：OpenCV 觸發是否需要 dwell time（隱性未明寫）
- **選項 A**：偵測到人即觸發（規格目前隱含）
- **選項 B**：相機框內持續偵測到人 ≥ 1.5s 才觸發
- **選項 C**：偵測到人 + 距離店面 < N 公尺（需深度資訊）
- **🌟 推薦**：**選項 B**。理由：(1) 防 80% 誤觸（路人、影子、商家走過）；(2) S6+ 真實 OpenCV 易實作；(3) S1 v2 stub 階段不影響（按 p 模擬即可）。建議規格現在就加註：「OpenCV 觸發定義 = 相機框內偵測到人持續 ≥ 1.5s」。

### 點 5-15：L4 客服模式內按 `s` 應該怎麼處理（隱性 — 邊界）
- **選項 A**：忽略 `s`，走「請選擇 1/2」重複提示（規格目前隱含）

- **🌟 推薦**：**選項 B**。理由：(1) 解決 B4-3 嚴重 bug；(2) 顧客在客服模式內按 `s` 的真實意圖 100% 是「我搞定了、要付款了」；(3) 規格鏈路 A 的 trigger 是「終端輸入 `s`」— 應該在 L4 全狀態（含客服模式內）都優先處理。
