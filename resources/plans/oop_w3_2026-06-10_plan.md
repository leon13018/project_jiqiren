## Commit 1 — TimedConfirm 家族 + 測試

1. **[RED]** 新增 `tests/sales/test_timed_confirm.py`，三組：
   - **骨架測試**（測試內自訂 micro 子類別，`timeout` 用 0.01~0.05 小值，read 用 scripted lambda，**禁 sleep**）：
     a. hook 順序：on_enter 先於 prompt 的 speak_blocking（事件記錄 list 驗序）
     b. classify 回非 None 字串 → 立即返回該值
     c. **classify 回 `False` → 立即返回 `False`**（is-not-None 語意，不得被當亂答續迴圈）
     d. classify 回 None → on_unclear 被呼叫、迴圈續跑，下一個 response 命中即返回
     e. read 回 None（silent）→ 返回 on_timeout 結果
     f. 亂答不重置 deadline：micro timeout + scripted 連續亂答 → 最終返回 on_timeout 結果
   - **單例 config 接線**：`CANCEL_CONFIRM.prompt is CANCEL_CONFIRM_PROMPT`、`.timeout == CANCEL_CONFIRM_TIMEOUT`；`SERVICE_CONFIRM.allow_scan is False`、`SERVICE_CONFIRM_SCAN.allow_scan is True`、timeout == `L4_C_CONFIRM_TIMEOUT`；`INVALID_QTY_CANCEL_CONFIRM.timeout == INVALID_QTY_CANCEL_CONFIRM_TIMEOUT`
   - **行為抽查**（scripted DialogIO）：CancelConfirm「不要取消」→False /「取消」→True / silent→True；SERVICE_CONFIRM_SCAN 收 "s"→"scan"；SERVICE_CONFIRM 收 "s"→走亂答（io.speak 收到 `L4_UNCLEAR_NOTICE`）後下一 response「好」→"yes"；ServiceConfirm on_enter 印 `SERVICE_PHONE`；InvalidQtyCancelConfirm 亂答 → speak_blocking 收到 `INVALID_QTY_UNCLEAR_PREFIX` 開頭重播、「退出」→"exit"、silent→"cancel_overlimit"
